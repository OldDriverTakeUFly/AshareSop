# 语音化转换规则（speech-conversion）

研报是给「看」设计的——密集表格、对照矩阵、ASCII 框图、百分位数字——直接朗读（TTS）时无法听懂。本文档定义把 md 研报转为「适合耳朵听」的口语化文本的完整规则，由 `scripts/report_to_speech_text.py` 实现。

> 本文档是 `SKILL.md` 步骤 24「语音化」的参考细则。**写作研报时不必遵循本文档**，只需保证 md 报告质量；语音化是写作完成后的独立转换步骤。

---

## 1. 为什么不能直接读 md

| md 中的元素 | 读出来的效果 | 问题 |
|-------------|-------------|------|
| 9×7 财务表 | "竖线 13.97 竖线 负15.56% 竖线..." 连续 9 行 | 流水账，听者记不住也无法对比 |
| 对照矩阵表（A vs B 8 维度） | 逐行读列值 | 丢失"谁强谁弱"的视觉锚点 |
| ASCII 框图（决策矩阵） | 乱码字符 | 完全无法理解 |
| `**加粗**` `[链接](url)` `### 标题` | 读出"星号星号"、URL 字符串 | 噪音 |
| "PE 85.7% 分位" | 裸数字 | 需解释"比历史上 85% 的时间都贵" |
| 一句话含 3+ 百分比 | "毛利率 36.04% ROE 9.93% 净利 +859.78%" | 听到后半句忘了前半句 |

**核心矛盾**：报告是给「看」设计的，而语音化要面向「听」。转换器必须把表格转成口语化叙述，而非逐行流水账。

---

## 2. 三层架构

`scripts/report_to_speech_text.py` 采用**规则层 + LLM 层 + 降级**的三层架构。规则层处理所有确定性工作（免费、可重复），LLM 层只处理需要语义理解的"硬骨头"，LLM 不可用时自动降级到规则层。

```
输入 md
  │
  ├─ ① 规则层（确定性，零成本）
  │   ├─ 剥离 frontmatter / 代码块 / HTML 注释
  │   ├─ 识别并丢弃 ASCII 框图（┌─┐│ 等框线字符行）
  │   ├─ 跳过非内容段（见 §3）
  │   ├─ 去 markdown 标记（** [链接] `code` # > - emoji）
  │   ├─ 丢弃纯 URL 行
  │   └─ 段落切分 + 表格块识别（连续 | 行视为一个 table block）
  │        ↓
  ├─ ② LLM 层（按需，调 GLM glm-4-flash）
  │   仅对两类"硬骨头"调用：
  │   ├─ 表格块 → 「取首尾/极值 + 表头语义 → 口语化叙述」
  │   └─ 数字密集段（同段 ≥3 个百分比/大数）→ 口语化重写
  │   每次调用后做数字一致性校验（见 §5），校验失败则降级。
  │        ↓
  └─ ③ 降级（LLM 不可用 / 失败 / --no-llm / 数字校验失败）
      规则键值叙述：取表头+首行+末行，拼成"起始：键值；最新：键值"句式。
      不如 LLM 自然，但零依赖、可重复、离线可用。
  ↓
输出 docs_speech/.../<name>.txt
```

## 3. 跳过的非内容段

以下标题关键词命中时，**整段（直到下一个同级标题）被剔除**，不出现在语音版中：

- 免责声明
- 版本历史
- 附录
- 数据源索引 / 数据源与计算方法 / 数据源清单 / 关键计算说明 / 引擎调用记录 / 横向市场估值锚定
- Source of Truth
- Ultraworked with / Co-authored-by（agent 署名 footer）

这些段对「看」有价值（溯源、合规），但对「听」是噪音。

> **如需新增跳过段**：编辑 `scripts/report_to_speech_text.py` 顶部的 `SKIP_SECTIONS` 集合。

## 4. 表格叙述化策略（核心）

### 4.1 LLM 叙述（默认，推荐）

LLM 收到表格的 markdown 原文，被要求：
1. **只用表格中真实出现的数字**，禁止计算/估算/四舍五入/编造
2. 抓**首尾值、极值、拐点**，不必逐行罗列
3. 用表头语义解释数字（"净利润"而非裸数）
4. 中文口语，1-3 句话，禁用 markdown

**转换示例**（锦浪科技财务表）：
```
原表格（9行）：
  2024Q1  营收13.97(-15.56%)  净利0.20(-93.74%)
  2025Q1  营收15.18(+8.65%)   净利1.95(+859.78%)
  2026Q1  营收14.19(-6.51%)   净利0.61(-68.59%)

LLM 输出：
  2024年第一季度营收13.97亿、同比下滑15.56%，净利润仅0.20亿、同比剧降93.74%；
  到2025年一季度大幅反弹至1.95亿、同比暴增859.78%；然而2026年一季度再次回落
  至0.61亿、同比减少68.59%。
```

### 4.2 规则降级（LLM 不可用时）

取表头 + 首行 + 末行（行数 ≤5 时全报），拼成键值句式：
```
起始：报告期2024Q1，营收13.97，同比-15.56%，归母净利0.20，同比-93.74%；
最新：报告期2026Q1，营收14.19，同比-6.51%，归母净利0.61，同比-68.59%。
```
信息损失较多（无趋势归纳），但保证零依赖可用。

### 4.3 不同表格类型的处理倾向

| 表格类型 | LLM 叙述效果 | 说明 |
|---------|-------------|------|
| 时间序列表（财务季度数据） | ⭐⭐⭐ 优秀 | 首尾/拐点天然适合"从 X 到 Y"叙事 |
| 对比表（A vs B 多维度） | ⭐⭐⭐ 优秀 | LLM 能归纳"谁强谁弱" |
| 情景区间表（悲观/中性/乐观） | ⭐⭐⭐ 优秀 | LLM 能描述"跨度大、分歧明显" |
| 排名榜（涨停/资金流排行） | ⭐⭐ 中等 | 首尾无趋势意义，LLM 倾向报极值+定性 |
| 清单表（个股列表） | ⭐ 一般 | 逐个报冗长，LLM 会挑重点，但信息损失大 |

## 5. 防幻觉：数字一致性校验

LLM 偶尔会改写数字（如把 `13.97` 误写成 `13.79`）。脚本对每次 LLM 输出做校验：

- 提取源文本和输出文本中的所有数字（按**绝对值**比较——口语化常把 `-15.56%` 说成"下降 15.56%"，去掉负号是合理转换，不应判为编造）
- 输出中出现源文本没有的数字 → 判定疑似编造 → **降级为规则叙述**

校验在 `scripts/report_to_speech_text.py` 的 `_verify_numbers()` 实现。

> **已知局限**：该校验只防"凭空编造数字"，不防"数字正确但张冠李戴"（如把营收值安到净利上）。LLM prompt 已强约束用表头语义，但极端情况下仍可能出错——关键数字建议人工复核。

## 6. 用法

### 6.1 前置配置（.env）

语音化的 LLM 层只需一个配置（已 gitignore，本地配置）：

```bash
LLM_API_KEY=<你的智谱 API key>
```

以下两项**通常无需配置**，脚本会自动处理：

- **`LLM_BASE_URL`**：脚本自动探测 Coding Plan 端点（`api/coding/paas/v4`）与按量付费端点（`api/paas/v4`），选用可用的那个。Coding Plan 用户即使误配按量付费端点也会自动切换。
- **`LLM_MODEL`**：脚本检测到推理模型（glm-5.x/glm-4.5，会把 token 花在 reasoning 上导致 content 为空）时，自动切换为非推理模型 `glm-4-flash`（适合改写任务）。

### 6.2 命令

```bash
# 默认（LLM 叙述化，镜像输出到 docs_speech/）
.venv/bin/python scripts/report_to_speech_text.py "docs/盘后总结/2026-06-26_盘后总结.md"

# 关闭 LLM，纯规则降级（离线/省成本）
.venv/bin/python scripts/report_to_speech_text.py report.md --no-llm

# 指定输出路径
.venv/bin/python scripts/report_to_speech_text.py report.md --output /tmp/x.txt
```

### 6.3 产物

- **位置**：`docs_speech/...`（镜像 `docs/` 子目录结构，如 `docs_speech/盘后总结/2026-06-26_盘后总结.txt`）
- **格式**：纯文本（.txt），段落间空行分隔，无任何 markdown 标记
- **不入 git**：`docs_speech/` 已在 `.gitignore`，仅本地使用

### 6.4 与 report_to_audio.py 的关系

两步管线，可串联：

```
md 报告 ──report_to_speech_text.py──▶ docs_speech/*.txt（适合朗读的文本）
                                              │
                                              ▼
                                   report_to_audio.py ──▶ docs_audio/*.mp3
```

若要让 MP3 也是口语化版本，先用 `report_to_speech_text.py` 生成 .txt，再用 `report_to_audio.py` 把该 .txt 转 MP3（report_to_audio 接受任意文本输入）。但注意 `report_to_audio.py` 内置的 `clean_markdown()` 会对 .txt 再做一次清理——传入 .txt 时其 markdown 标记已剥离，清理基本是 no-op，安全。

## 7. 自动处理与剩余注意

### 7.1 已自动处理（无需用户干预）

以下情形脚本已从代码层面解决，用户不必手动配置或排查：

| 情形 | 脚本的自动处理 |
|------|---------------|
| `PROJECT_ROOT` 指向无效路径（如 Docker 遗留 `/app`） | `stockhot/core/config.py` 的目录创建已容错——import 时只 warn 不崩，真正用到 DB/文件时才报清晰错误 |
| shell 设了 `ALL_PROXY=socks://` | 脚本启动时自动清除 socks 代理变量（openai/httpx 不支持 socks scheme），保留 HTTP/HTTPS_PROXY |
| Coding Plan 用户误用按量付费端点（报 1113 余额不足） | 脚本自动探测并切换到 Coding Plan 端点 `api/coding/paas/v4` |
| 配了推理模型（glm-5.x/glm-4.5，content 返回空） | 脚本自动切换为非推理模型 `glm-4-flash` |

### 7.2 剩余注意（属正常机制或需用户配置）

| 现象 | 说明 | 处理 |
|------|------|------|
| LLM 全程降级为规则版 | `.env` 未配 `LLM_API_KEY`，或网络不通 | 配置 key 后重试；或直接接受规则版（仍可读） |
| `[LLM 校验] 检测到疑似编造数字` 后降级 | LLM 输出含源文本没有的数字，防幻觉机制触发 | 正常机制，已自动降级规则叙述。若某表格频繁触发，说明该表过于复杂，规则版兜底是合理的 |
| 个别表格降级为规则版 | 该表格 LLM 叙述失败或数字校验未过 | 正常现象——36 表格中降级 1-2 个属预期，规则版仍可读 |

## 8. 实测性能（参考）

| 报告类型 | 表格数 | LLM 成功率 | 耗时 |
|---------|--------|-----------|------|
| 盘后总结 | 6 | 6/6 | ~35s |
| 个股深度研报 | 36 | 45/46（1 降级） | ~3min |
| 盘前整理 | 2 | 3/3 | ~15s |

单次 LLM 调用约 3 秒（glm-4-flash）。大报告（>30 表格）耗时会到分钟级，运行时有进度输出（`[LLM n] 表格叙述化...`）。
