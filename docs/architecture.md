# 架构设计

本文档说明 CodeAgent Dashboard 的整体工程架构，重点解释为什么选择模块化单体，以及各个模块如何围绕任务、执行、审批和分析形成一个可落地的系统。

## 1. 架构目标

项目的第一阶段需要在保证实现速度的同时，保持清晰边界和可演进性。因此选择模块化单体，而不是一开始拆成多个微服务。

这样做的原因有三点：

1. 当前领域边界虽然明确，但业务还在快速收敛，过早拆分会增加通信、部署和一致性成本。
2. MVP 的价值在于验证任务闭环，而不是验证分布式基础设施。
3. 模块化单体足以在代码层面建立清楚边界，后续若某些模块压力明显上升，再考虑抽离。

## 2. 总体架构风格

建议采用前后端分离的 Web 应用形态：

- 前端提供 Dashboard、任务页、运行详情页、审批页、分析页
- 后端在单一部署单元内承载多个业务模块
- 数据层保存任务、运行、审批、审计、Agent 与分析数据
- 后台作业或异步执行机制负责调度 Agent 和聚合分析

逻辑上可分为以下层次：

- Presentation Layer
- Application Layer
- Domain Layer
- Infrastructure Layer

这四层是工程组织方式，不要求必须落成严格的技术框架，但必须保持依赖方向稳定。

## 3. 核心模块划分

### 3.1 Task 模块

职责：

- 管理任务生命周期
- 提供任务创建、更新、查询能力
- 管理任务层级关系
- 汇总任务视图状态

Task 模块关心的是“要做什么”，不是“这次是怎么跑的”。

在 MVP 中，若一个任务需要拆分并行处理，Task 模块负责创建和管理子 `Task`。也就是说，子任务是业务层对象，而不是仅存在于运行层的临时分支。

### 3.2 TaskRun 模块

职责：

- 创建与管理运行记录
- 跟踪执行状态变化
- 保存运行输出摘要和失败原因
- 支撑重试与执行历史查询

TaskRun 模块关心的是“具体某次执行发生了什么”。

MVP 中，一个 `TaskRun` 只对应一个主执行 Agent。多个 Agent 的并行能力通过多个子 `Task` 各自创建 `TaskRun` 来实现，而不是让多个 Agent 写入同一个运行实例。

### 3.3 Approval 模块

职责：

- 创建审批请求
- 处理批准与拒绝
- 向任务与运行状态反馈审批结果
- 提供待审批列表和审批历史

审批要作为独立模块存在，这样规则才能集中管理，而不是散落在各个页面或执行分支里。

### 3.4 Audit 模块

职责：

- 记录关键业务事件
- 提供时间线查询能力
- 为问题回溯和行为解释提供依据

审计不是普通日志收集。它需要围绕业务事件建模，支持按任务、运行、审批等对象查询。

### 3.5 Orchestrator 模块

职责：

- 接收任务执行请求
- 决定串行还是并行执行
- 选择 Agent
- 创建并管理 `TaskRun`
- 汇总子任务或子运行的完成情况

这是系统的执行中枢。MVP 中它不必非常智能，但必须足够稳定和可解释。

### 3.6 Agent Registry 模块

职责：

- 管理可用 Agent 列表
- 维护 Agent 能力、状态、并发上限
- 提供可执行 Agent 查询接口

这个模块把“有哪些 Agent 可用”从调度逻辑中独立出来，便于未来替换、扩展和治理。

### 3.7 Analytics 模块

职责：

- 聚合任务与运行指标
- 输出 Dashboard 所需分析数据
- 支撑 Agent 表现评估
- 为流程优化提供观察基础

Analytics 不只是报表层，它会反过来影响后续产品决策，例如是否值得继续增强并行策略。

## 4. 关键交互流程

### 4.1 创建任务

1. 前端提交任务表单
2. Task 模块校验并创建 `Task`
3. Audit 模块记录任务创建事件
4. 若配置为立即执行，调用 Orchestrator

### 4.2 启动执行

1. Orchestrator 读取任务信息
2. 判断是否拆分子任务
3. 查询 Agent Registry 获取匹配 Agent
4. 创建一个或多个 `TaskRun`
5. 派发执行请求
6. Audit 模块记录调度与派发事件

### 4.3 并行执行

MVP 架构明确支持并行执行独立任务或子任务。推荐方式是由 Orchestrator 根据依赖关系生成多个可并发子 `Task`，并分别为它们创建运行记录。

并行执行的规则应尽量简单：

- 只有无依赖关系的节点可并发
- 每个分支单独选择 Agent
- 每个分支单独汇报状态
- 汇总节点等待必要分支完成

这里有一个需要明确的限制：MVP 里的并行分支是彼此独立的工作项，不共享可变上下文，也不进行 Agent 间协商或自动接力。

为什么在 MVP 就支持这点，因为它对用户价值直接且明显。只要任务可拆，整体耗时就有机会下降。相比之下，复杂多 Agent 协作虽然更先进，但对第一版不是必要条件。

### 4.4 审批介入

MVP 默认采用“最终结果验收审批”作为主审批链路：

1. 某个运行成功结束且任务被标记为需要审批
2. Approval 模块创建 `Approval`
3. Task 或 TaskRun 状态进入 `waiting_approval`
4. 审批者处理后，结果返回给 Orchestrator 或 TaskRun 模块
5. 审批通过则工作项完成，审批拒绝则回到可重新处理状态
6. 审计模块记录审批事件

### 4.5 分析聚合

1. Task、TaskRun、Approval、Audit 产生事件或数据变化
2. Analytics 模块周期性或实时聚合指标
3. Dashboard 查询 Analytics 读模型

## 5. 数据与一致性策略

在模块化单体中，模块共享同一个数据库实例是合理的，但不代表可以随意跨模块读写表。建议遵循以下原则：

- 每个模块拥有自己的数据访问入口
- 跨模块调用优先通过应用服务或领域事件完成
- 审计事件在关键状态变更时同步写入
- Analytics 可采用异步聚合，接受短暂延迟

任务总体状态的收敛规则应由后端集中计算。前端只能消费任务状态结果，不应自行根据运行列表和审批列表拼装推断。

MVP 不必追求所有分析数据实时更新。只要主业务链路一致，分析层允许有可接受的延迟。

## 6. 模块边界建议

可以按如下目录思路组织代码，目的是表达边界，不是强制某种框架：

```text
src/
  modules/
    task/
    task-run/
    approval/
    audit/
    orchestrator/
    agent-registry/
    analytics/
  shared/
  app/
```

每个模块至少包含：

- domain
- application
- infrastructure
- api 或 handlers

这样能在一个代码库内维持相对清楚的内聚关系。

## 7. 关于多 Agent 协作的架构定位

需要明确区分两层能力：

### 7.1 MVP 已经要做的

- 独立任务并行调度
- 独立子任务并行调度
- Agent 选择与并发控制
- 运行结果汇总
- 根任务、子任务、运行记录之间的清晰映射

### 7.2 v1 再做的

- Agent 之间直接交换中间成果
- 多 Agent 协商或评审
- 动态重规划与角色切换
- 面向复杂依赖图的高级编排

所以系统架构要从一开始就预留扩展点，但不应在 MVP 中引入复杂通信协议或自治机制。

## 8. 可观测性建议

为了让系统易于调试和运营，建议从第一版就具备基础可观测性：

- 关键模块结构化日志
- 以 `correlationId` 关联一次任务执行链路
- 运行失败和审批等待的告警基础能力
- 面向 Dashboard 的系统健康指标

这部分与业务审计不同，但两者可以互补。可观测性帮助排障，审计帮助解释业务过程。

## 9. 架构结论

这套架构的关键不是技术炫技，而是让产品目标可以稳定落地。模块化单体适合当前阶段，`Task` 与 `TaskRun` 分离保证语义清楚，Orchestrator 和 Agent Registry 负责执行，Approval 与 Audit 负责可控和可追溯，Analytics 负责形成持续优化闭环。并行执行在 MVP 中就值得落地，但范围只限独立任务与独立子任务，复杂多 Agent 协作留到 v1 更合适。
