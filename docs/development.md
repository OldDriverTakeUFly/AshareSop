# 开发说明

本文档给出 CodeAgent Dashboard 的工程实施约定，目标是让这套文档不仅能指导“做什么”，也能指导“怎么开始做”。这里不规定过多细节，而是聚焦对 MVP 落地最重要的开发原则。

## 1. 开发目标

开发工作的第一优先级是把 MVP 文档中的核心闭环做出来，而不是过早追求通用平台能力。工程实现应始终围绕以下问题：

- 能否清楚管理 `Task`
- 能否稳定创建和追踪 `TaskRun`
- 能否在关键节点处理 `Approval`
- 能否保留完整 `AuditEvent`
- 能否支持独立任务或子任务并行执行
- 能否输出对产品有意义的基础 Analytics

如果某项技术设计不能明显帮助上述目标，就不应该优先进入 MVP。

## 2. 开发原则

### 2.1 先保语义清楚，再保抽象漂亮

`Task`、`TaskRun`、`Approval`、`AuditEvent`、`Agent` 这些概念必须在代码、接口和界面中保持稳定含义。命名不清会直接导致实现偏离文档。

### 2.2 模块内聚，模块间显式协作

虽然项目是模块化单体，但模块之间不能随意越界读写。跨模块行为应通过明确的 service、use case 或事件触发来完成。

对 MVP 来说，还需要额外统一一个建模约束：**子任务用子 `Task` 表达，运行层只记录执行事实，不负责承载主业务层的任务树。**

### 2.3 审计优先于便利

任何会改变任务执行事实的动作，都应优先考虑是否需要产生审计事件。尤其是状态流转、分派、审批和失败重试。

### 2.4 并行能力要受控

MVP 需要支持并行执行，但只面向独立任务和独立子任务。实现时应先保证以下几点：

- 并发上限可控
- 每个并行分支都有独立记录
- 汇总逻辑能正确识别完成与失败
- 用户能看懂每个分支在做什么

这里的“独立记录”具体指：每个并行子任务拥有自己的 `Task` 与 `TaskRun`；不要在 MVP 中依赖嵌套 `TaskRun` 去表达主要业务层级。

复杂多 Agent 协作不属于当前开发目标，避免为未来可能的能力提前堆砌复杂机制。

第一版也不实现 Agent 间共享记忆、自动接力、协商式决策。这些能力只有在独立并行调度被证明稳定之后再考虑。

## 3. 推荐实施顺序

为了尽快形成可演示、可验证的系统，建议按下面的顺序推进。

### 阶段 1，打通最小任务链路

- 建立 Task 模块
- 支持任务创建、列表、详情
- 建立 TaskRun 模块的基础状态机
- 能从任务触发一次运行

这个阶段就应把任务总体状态的收敛规则写进后端，而不是等前端页面做好后再补。

### 阶段 2，加入 Agent 与调度

- 建立 Agent Registry
- 支持按能力筛选 Agent
- 建立基础 Orchestrator
- 支持单任务执行与状态回传

### 阶段 3，加入审批与审计

- 建立 Approval 模块
- 在最终结果验收节点插入审批
- 建立 Audit 模块和时间线展示

### 阶段 4，加入并行执行

- 支持独立任务并发运行
- 支持一个任务拆成独立子任务后并发运行
- 支持运行汇总和失败处理

推荐把“拆分成子任务”实现为明确的子 `Task` 创建流程，而不是临时内存结构。这样前端、审计和分析才能共享同一套事实来源。

### 阶段 5，加入 Analytics 与打磨 Dashboard

- 输出基础统计指标
- 优化 Dashboard 视图
- 让用户能从总览页快速定位问题和审批项

这种顺序能确保团队尽早看到真正工作的系统，而不是只完成一些孤立模块。

## 4. 前后端协作建议

### 前端重点

- 让任务、运行、审批、审计四类信息在界面上容易区分
- 优先保证状态展示准确
- 不要在前端自行推断复杂业务状态，以后端返回为准

### 后端重点

- 把状态流转规则收敛到模块内部
- 保证关键写操作具备审计记录
- 为 Dashboard 提供稳定、面向页面的查询接口

### 共同原则

- 界面文案与领域术语统一
- 列表页与详情页对同一状态的解释一致
- 并行执行的展示和后端模型保持一一对应

## 5. 测试策略

MVP 阶段建议采用分层测试，而不是把所有验证都压到端到端测试。

### 5.1 单元测试

重点覆盖：

- 任务状态规则
- 运行状态流转
- 审批决策规则
- 调度器对依赖和并行性的判断
- 根任务在多个子任务结果下的状态收敛规则

### 5.2 集成测试

重点覆盖：

- 任务创建后触发运行
- 运行进入审批后再继续
- 失败重试生成新的 `TaskRun`
- 并行子任务完成后正确汇总
- 审批拒绝后任务回到可重新处理状态

### 5.3 最少端到端验证

至少验证以下主流程：

1. 创建任务
2. 启动执行
3. 查看运行状态
4. 处理审批
5. 查看任务完成和审计轨迹

## 6. 数据与迁移建议

虽然当前仓库还是空白项目，但从一开始就建议把数据库变更纳入版本管理。任何领域对象结构调整，都应同步更新迁移脚本和文档。

需要特别注意两点：

- 不要把运行历史设计成可覆盖结构
- 不要为了省表而把审批或审计揉进任务主表

这两种做法都会让后续分析和追踪变得困难。

## 7. API 设计建议

接口设计应尽量围绕业务动作，而不是仅围绕数据表。

示例方向：

- `POST /tasks`
- `GET /tasks`
- `GET /tasks/:id`
- `POST /tasks/:id/runs`
- `GET /tasks/:id/runs`
- `POST /approvals/:id/approve`
- `POST /approvals/:id/reject`
- `GET /dashboard/summary`

这里的接口只是建议命名方向，实际实现时可以调整，但语义要稳定。

## 8. 并行执行开发要求

这是本项目 MVP 的一个关键点，需要明确写入开发约定。

### 8.1 必须支持的场景

- 多个独立任务同时执行
- 一个任务下多个独立子任务同时执行

### 8.2 必须满足的行为

- 每个并行分支都可追踪
- 每个分支都能关联具体 Agent
- 分支失败不会静默丢失
- 汇总结果能反映整体是否完成
- 并行分支之间没有共享记忆和自动协商假设

### 8.3 暂不处理的复杂能力

- Agent 之间共享复杂上下文
- 多 Agent 自动协商计划
- 并行过程中动态改写整个任务图

推荐判断标准是，MVP 先把“能并行做”做稳，v1 再把“能协作做”做深。

## 10. 开发结论

本项目的实现重点很明确，先建立稳定的任务执行工作台，把任务、运行、审批、审计、调度和分析串成一个最小闭环。工程上采用模块化单体，产品上强调可追踪与可控，执行上支持独立任务的并行加速。这样既能让 MVP 尽快成型，也能为 v1 的更复杂多 Agent 协作留下清楚的扩展空间。
