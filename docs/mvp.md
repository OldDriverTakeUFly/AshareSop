# MVP 定义

本文档定义 CodeAgent Dashboard 的最小可用版本，重点回答三件事：第一版必须解决什么问题，必须交付哪些能力，以及哪些能力明确不在 MVP 范围内。

## 1. MVP 目标

MVP 要完成的不是一个概念验证页面，而是一个可实际使用的任务执行看板。用户应能在系统内完成从任务创建到执行追踪的最小闭环，并在关键风险点进行人工审批。

MVP 重点验证以下假设：

1. 用户愿意在统一面板内管理任务和 AI Agent 执行过程。
2. `Task` 与 `TaskRun` 分离后，任务管理和执行分析会更清晰。
3. 对独立任务或独立子任务启用并行 Agent 执行，能明显缩短交付时间。
4. 审批和审计能力能提高用户对 AI 执行系统的信任度。

## 2. 目标用户

MVP 优先服务两类角色：

### 2.1 任务发起者

通常是产品经理、研发负责人、项目协调人，负责：

- 创建任务
- 设定优先级和期望结果
- 选择是否立即执行
- 查看整体进度和结果

### 2.2 审批者或监督者

通常是资深工程师、团队负责人、项目 owner，负责：

- 审核高风险动作
- 决定是否接受执行结果
- 处理失败、重试、重新分派等情况

MVP 也会隐含第三类系统角色，即 AI Agent。Agent 不是传统登录用户，但它是系统中的执行主体，需要被注册、分派、衡量和追踪。

## 3. MVP 用户故事

### 3.1 创建并执行任务

用户创建一个任务，填写标题、描述、优先级、预期输出、是否可拆分、是否需要审批。创建后，可以选择立即执行或稍后执行。

### 3.2 查看进度

用户进入 Dashboard，能快速看到：

- 总任务数和不同状态分布
- 当前执行中的任务和子任务
- 待审批事项
- 最近失败的运行

### 3.3 分派给 Agent

系统根据任务配置和 Agent 能力，从 Agent Registry 中选择一个可用 Agent 来执行当前工作项。若一个任务被拆成多个独立子任务，则系统可以为不同子任务分别选择 Agent，并并行执行。

### 3.4 审批结果

当某个任务运行进入待审批状态时，审批者可以查看上下文、输出摘要和审计事件，然后批准继续、批准完成，或拒绝并要求重新执行。

### 3.5 查看执行历史

用户能查看某个任务关联的所有 `TaskRun`，知道它被执行过几次、分别由谁执行、何时成功、何时失败、失败原因是什么。

## 4. MVP 范围

### 4.1 必做能力

#### A. Dashboard 总览

- 任务总数
- 状态分布
- 运行中任务列表
- 待审批列表
- 最近执行活动
- 基础分析卡片

#### B. 任务管理

- 创建任务
- 编辑任务基础信息
- 查看任务详情
- 手动触发执行
- 取消未开始或可安全取消的任务

#### C. 执行管理

- 为任务创建 `TaskRun`
- 记录运行状态变化
- 支持失败重试
- 展示运行详情和输出摘要

#### D. 审批流

- 创建审批项
- 展示待审批内容
- 批准或拒绝
- 审批动作进入审计轨迹

MVP 默认只要求支持一种主审批模式：**最终结果验收审批**。也就是说，任务或子任务执行完成后，如果该工作项要求审批，则进入待审批状态；审批通过后完成，审批拒绝后回到可重新处理状态。

开始前审批、执行中高风险动作审批可以在架构上预留，但不要求作为第一版必须交付的完整产品能力。

#### E. 审计轨迹

- 记录任务创建、更新、调度、开始执行、结束执行、失败、审批等事件
- 在任务详情和运行详情中展示时间线

#### F. Agent Registry

- 注册 Agent
- 标记 Agent 的能力、状态和可接收任务类型
- 根据能力筛选可执行 Agent

#### G. Analytics

- 任务完成率
- 运行成功率
- 平均执行时长
- 平均审批等待时长
- 按 Agent 的基础表现统计

### 4.2 明确支持的并行执行能力

MVP 应支持有限但实用的并行执行。

支持范围如下：

- 多个彼此独立的任务可以并行执行
- 一个任务下若存在互不依赖的子任务，可以分派给多个 Agent 并行执行
- 在 MVP 中，子任务采用子 `Task` 建模，而不是多个 Agent 共享同一个 `TaskRun`
- 每个并行分支都应拥有独立的 `Task` 与 `TaskRun` 记录，并进入统一审计轨迹

推荐原因很直接，很多实际工作天然可拆分，例如：

- 一个需求拆成文档说明、技术调研、测试设计
- 一个问题拆成日志分析、影响面排查、修复方案草拟
- 一个交付拆成前端实现说明、后端接口草案、验证清单

MVP 不要求实现复杂的多 Agent 协商系统，只要能稳定处理独立工作项并行即可。这样既能提升速度，也不会把第一版系统复杂度推得过高。

这里的边界要写清楚：MVP 支持 **independent parallelism**，不支持共享记忆、动态协商、自动接力、联合决策这类 **collaborative multi-agent** 能力。

### 4.3 明确不在 MVP 的能力

以下内容放到后续版本：

- 多 Agent 自动协商与角色辩论
- 高级计划生成与动态任务树重写
- 基于长期记忆的复杂上下文共享
- 细粒度权限系统和组织级治理
- 跨项目视图和跨团队容量规划
- 外部平台深度集成，如代码仓库、IM、工单系统的大规模联动

## 5. 关键流程

### 5.1 任务创建到完成

1. 用户创建 `Task`
2. 系统校验任务字段并保存
3. 用户手动触发或系统自动触发执行
4. Orchestrator 创建一个或多个 `TaskRun`
5. Agent 执行并上报状态
6. 如需审批，系统创建 `Approval`
7. 审批通过后任务继续或完成
8. 任务和运行结果进入 Analytics 聚合

### 5.2 失败与重试

1. `TaskRun` 失败
2. 系统记录失败原因和上下文
3. 用户查看失败详情
4. 用户选择重试、改派 Agent，或关闭任务
5. 新一轮执行生成新的 `TaskRun`

### 5.3 并行子任务执行

1. 用户或系统将一个根任务拆成多个子 `Task`
2. 每个子 `Task` 继承必要上下文并单独进入可执行状态
3. Orchestrator 为每个子 `Task` 匹配合适 Agent
4. 每个子 `Task` 创建独立 `TaskRun` 并并发执行
5. 系统分别记录状态、输出和审计事件
6. 全部必要子任务结束后，根任务汇总结果，若需要则进入统一审批

## 6. MVP 成功标准

MVP 完成后，至少应能证明：

- 用户可以独立完成从建任务到看结果的全流程
- 任务与运行记录可以清楚分开查看
- 审批点和审计轨迹足够支撑回溯
- 并行执行在独立任务场景下能够稳定工作
- 分析数据足够反映系统基础健康度

## 7. v1 扩展方向

v1 可以在 MVP 稳定后补强多 Agent 协作能力，但原则是建立在现有可控调度之上。推荐扩展方向包括：

- 支持更复杂的任务依赖图
- 支持 Agent 之间的上下文传递
- 支持一个 Agent 产出由另一个 Agent 继续加工
- 支持更智能的 Agent 选择和负载分配
- 支持对并行执行结果做自动汇总和冲突检测

总结来说，MVP 的并行策略是，先把独立项并行做好。v1 再做协作型多 Agent 系统。
