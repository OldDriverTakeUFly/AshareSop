# 本地开发环境管理 Skill 说明

这份文档是 `skills/local-development-environment/` 的中文说明版，面向人类读者。它的目标不是替代 `SKILL.md`，而是帮助团队快速理解这套 skill 想解决什么问题、适合什么时候使用，以及为什么它能减少本地依赖互相污染。

真正的规范性内容仍以 `SKILL.md` 为准。如果 agent 需要执行环境相关工作，应优先读取 `SKILL.md` 和配套 checklist、examples、reference 文件。

## 这套 skill 解决什么问题

很多本地开发环境问题，本质上都不是“工具不够多”，而是环境边界不清晰。常见表现包括：

- 一个项目安装的 Python 依赖影响了另一个项目
- Node 版本切换混乱，项目 A 能跑，项目 B 不能跑
- PostgreSQL、Redis 这类服务直接装在宿主机，久而久之没人知道哪个项目依赖什么
- shell 启动脚本里塞了很多临时修复，后面越来越难排查
- 项目里没有明确 runtime 版本和 lockfile，导致今天能跑、下周就坏

这套 skill 的核心目标，就是让 agent 在处理本地开发环境时遵守同一套最小原则：

1. 先检查，再修改
2. 尽量使用项目级隔离，而不是宿主机级补丁
3. 尽量避免全局安装项目依赖
4. 运行时、项目依赖、服务依赖要分层管理
5. 改完以后必须验证，并明确说明做了什么

## 适用场景

当遇到以下任务时，适合使用这套 skill：

- 初始化一个项目的本地开发环境
- 在多个项目之间切换，并避免 Python / Node / Java 版本互相影响
- 修复已经混乱的本地依赖环境
- 设计团队统一的本地开发环境规范
- 给仓库补 runtime pinning、lockfile、`.envrc` 或容器化本地服务方案
- 梳理“哪些东西应该全局安装，哪些应该项目内隔离”

## 核心原则

### 1. 先读现状，再决定怎么做

agent 不能一上来就推荐新工具，必须先检查仓库是否已经存在环境约定，例如：

- `.mise.toml`
- `.tool-versions`
- `.nvmrc`
- `pyproject.toml`
- `uv.lock`
- `package.json`
- `pnpm-lock.yaml`
- `docker-compose.yml`
- `.envrc`

如果项目已经有一套成型做法，应优先沿用，而不是轻易替换。

### 2. 优先项目级隔离

最稳妥的做法，不是把一切都装到宿主机，而是把环境边界尽量收缩到项目目录。

推荐思路是：

- runtime 用目录级版本管理
- Python 依赖进 `.venv`
- Node 依赖留在项目自己的 `node_modules`
- 数据库和中间件尽量走 `docker compose`

### 3. 避免把问题推给全局环境

这套 skill 明确反对把“项目依赖冲突”用“多装几个全局包”来解决。

例如，以下做法通常应避免：

- 全局 `pip install` 项目依赖
- 为了一个项目去改 shell 启动脚本
- 把本应写进项目文件的版本要求藏进个人机器配置
- 在同一个项目里随意混用多个包管理器

### 4. 分层看待依赖

这套 skill 要求 agent 把问题拆成三层：

- **运行时依赖**：Python、Node、Java、Ruby 等版本
- **项目依赖**：pip、uv、pnpm、npm 等安装的包
- **服务依赖**：PostgreSQL、Redis、Kafka、Elasticsearch 等本地服务

三层问题要分开处理，不要混成“环境坏了”这一个笼统结论。

### 5. 改完必须验证

环境调整不是“文件写好了就结束”。agent 必须验证：

- runtime 版本是否正确
- 包管理器是否按预期工作
- 项目的标准安装或启动命令是否可运行
- 本地服务是否真的能起来

## 推荐的默认思路

如果项目本身没有成熟方案，这套 skill 推荐一个尽量简单、可维护的默认组合：

- `mise` 或 `asdf` 管 runtime
- `uv` + `.venv` 管 Python
- `pnpm` 管 Node
- `direnv` 做目录级激活
- `docker compose` 跑数据库和中间件

注意，这不是要求每个项目都全量引入，而是作为“没有现成方案时的默认参考”。如果仓库已经有一套清晰且可用的环境约定，应优先遵循仓库现状与 `SKILL.md`，而不是被这份 README 里的默认组合覆盖。

## 给 agent 的默认行为预期

当 agent 使用这套 skill 时，应该做到：

- 在改动前说明自己检测到了什么
- 在多个备选方案中优先选“对现有仓库侵入最小”的方案
- 尽量不做静默的全局修改
- 改完后明确说明新增、复用、跳过了哪些内容
- 如果存在风险或未决项，必须说清楚

换句话说，这套 skill 不是为了让 agent 更激进，而是为了让 agent 在环境相关任务上更稳、更可追踪、更少污染宿主机。

## 如何使用这套 skill

如果你是人类开发者，可以把它当作团队的环境治理参考。

如果你要让不同 agent 都遵守同一套规则，建议：

1. 在仓库根目录放 `AGENTS.md`
2. 明确要求所有环境相关任务优先参考本 skill
3. 把 `SKILL.md` 视为唯一事实来源
4. 后续若要补语言专项规则，放在这个 skill 目录下扩展，而不是在多个地方复制规则

## 一句话总结

这套 skill 的本质，不是推广某个工具，而是建立一种稳定的本地开发环境治理方式：**项目内隔离优先、宿主机最小污染、修改前先检查、修改后必须验证。**
